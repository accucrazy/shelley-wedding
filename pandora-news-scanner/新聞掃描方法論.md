# 新聞稿媒體露出掃描 — 完整方法論

> 本文件整理自 2026 年 1–3 月全家＆7-ELEVEN 新聞掃描實戰經驗，涵蓋搜尋、篩選、驗證、IO 報表生成的完整流程與踩過的坑。
> 最後更新：2026-03-09 (v4.2 — 7-ELEVEN 補量+作者爬取+日期窗口過濾)

---

## 一、整體流程（v4.2 架構）

```
輸入：新聞稿標題 + 發稿日期（press_date）+ 關鍵字(8-13組) + 品牌詞 + --brand 參數
           ↓
Phase 1a：Google News RSS 搜尋（主力，含 after:press_date 日期限制）
           ↓
Phase 1b：DDG 廣域搜尋（max_results=80，前6組關鍵字 + 標題片段引號搜尋）
           ↓
Phase 1c：DDG site 補漏（300s timeout，容忍 5 次連續失敗，僅高價值媒體）
           ↓
硬性日期過濾：有日期的文章 < press_date → 直接淘汰
           ↓
預過濾：URL/標題規則排除非文章頁面（tag、category、首頁等）
           ↓
Phase 2：LLM 嚴格驗證（品牌動態 prompt，含日期意識，舊年份文章 → false）
           ↓
Phase 3：日期窗口過濾（距發稿日 >30 天 或 <0 天 → 移除）
           ↓
Phase 4：作者爬取（從文章 HTML 的 meta/JSON-LD 提取真實作者名）
           ↓
輸出：Google Sheet 月報 + JSON 備份
```

### 與舊版差異

| 項目 | 舊版 (v3) | v4.1 | v4.2（當前） |
|------|-----------|------|-------------|
| 搜尋策略 | DDG site:domain 逐媒體搜 | 三層搜尋 | 三層搜尋 + 放寬參數 |
| 日期控制 | 無 | GNews `after:` + 硬性過濾 + LLM | 同上 + 30天窗口過濾 |
| 媒體匹配 | 固定 20 家 | 345 家 domain map | 同上 |
| LLM 驗證 | 寬鬆 | 嚴格（品牌限定） | 品牌動態 prompt（BRAND_PROMPT_MAP） |
| 作者欄位 | 無 | 媒體名 fallback | HTML meta/JSON-LD 真實作者爬取 |
| 品牌支援 | 全家 only | 全家 only | 全家 + 7-ELEVEN（`--brand`） |
| DDG 參數 | max_results=50, kw[:4] | 同左 | max_results=80, kw[:6], timeout 300s |
| 有效結果(全家一月) | 195 筆 | 356 筆 (+83%) | 同上 |
| 7-11一月結果 | — | — | **465 筆**（313 頻道） |

## 二、搜尋來源（三層架構）

### 2.1 Phase 1a — Google News RSS（主力，56% 結果來源）

**優點**：提供可靠的發布日期、媒體名稱、覆蓋面廣，支援 `after:` 日期過濾
**缺點**：連結是 Google 轉址（`news.google.com/rss/articles/...`），無法直接抓取內文

```
URL 格式（v4.1 含日期限制）：
https://news.google.com/rss/search?q={關鍵字}+after:{press_date}&hl=zh-TW&gl=TW&ceid=TW:zh-Hant

回傳格式：XML RSS
每個 <item> 包含：
  - <title>：文章標題（含媒體名，如「標題 - ETtoday新聞雲」）
  - <link>：Google News 轉址連結
  - <pubDate>：發布日期（RFC 822 格式）
  - <source url="...">：媒體名稱 + 媒體 URL
```

**v4.1 改進**：
- 使用 `after:{press_date}` 參數直接在搜尋層面過濾舊文
- 從 `<source>` 標籤提取 `source_name`，作為媒體匹配的 fallback
- 對每組關鍵字 + 新聞稿標題片段都搜尋

### 2.2 Phase 1b — DDG 廣域搜尋（39% 結果來源）

**用法**：不加 `site:`，搜完再透過 domain map 匹配 345 家媒體

```python
from duckduckgo_search import DDGS

with DDGS() as ddgs:
    results = ddgs.text("全家草莓季", region="tw-tzh", max_results=80)
    # 每筆包含 title, href, body
    # 從 body 嘗試提取日期（正則匹配 YYYY-MM-DD 等格式）
```

**v4.2 改進**：
- `max_results` 從 50 提高到 **80**，增加覆蓋率
- 使用前 **6 組**關鍵字（原本 4 組）做 DDG 廣域搜尋
- 從 DDG snippet (body) 中提取日期，減少「無日期」的問題
- 有日期且早於 press_date → 直接淘汰（硬性過濾）

### 2.3 Phase 1c — DDG Site 補漏（5% 結果來源）

僅針對 Phase 1a+1b 未覆蓋的**高價值媒體**（約 80 家重點媒體）做 `site:domain` 搜尋。

**v4.2 設計理由與改進**：
- 舊版對全部 345 媒體逐一 DDG site 搜尋，耗時數小時且觸發 rate limit
- v4.1 只搜 Phase 1a+1b 漏掉的重點媒體，設 120 秒全域 timeout
- **v4.2**：timeout 從 120 秒提高至 **300 秒**，DDG 連續失敗容忍從 3 次提高至 **5 次**
- 7-ELEVEN 任務的關鍵字較分散，需要較長的搜尋時間才能充分覆蓋

### 2.4 搜尋策略總結

| 搜尋層 | 方法 | 目的 | 耗時 |
|--------|------|------|------|
| Phase 1a | GNews RSS + `after:` | 精準+日期可靠 | ~30s/任務 |
| Phase 1b | DDG broad + domain map | 補充覆蓋率 | ~20s/任務 |
| Phase 1c | DDG site（僅高價值） | 重點媒體補漏 | ≤300s/任務 |

### 3.2 關鍵字設計原則

| 原則 | 說明 |
|------|------|
| 必含品牌名 | 「全家」「7-ELEVEN」「統一超商」「7-11」 |
| 品牌名多版本覆蓋 | 7-ELEVEN 至少覆蓋 3 種寫法（7-ELEVEN / 7-11 / 統一超商） |
| 使用新聞稿中的專有名詞 | 「贅澤巧克力」「魚莓共生」「CT AMAZE」「翡翠莊園」 |

## 四、去重（URL 正規化）

### 4.1 正規化規則

```python
def normalize_url(url):
    url = url.split("#")[0].rstrip("/")              # 移除錨點和尾部斜線
    url = re.sub(r'[&?](utm_\w+|fbclid|gclid)=[^&]*', '', url)  # 移除追蹤參數
    url = re.sub(r'^https?://(www\.)?', '', url)     # 移除協議和 www
    return url.lower()
```

### 4.2 注意事項

- **不要移除所有 query 參數**！WordPress 網站用 `?p=12345` 作為文章識別符，移除會導致大量誤判為重複
- Google News 轉址 URL 各自唯一，不需特殊處理
- 同一篇文章在 AMP 版（`/amp/`）和一般版可能算重複，但保留兩者影響不大

## 五、硬性過濾（v4.1 多層日期防線）

### 5.1 三層日期過濾

v4.1 的核心改進在於**三層日期防線**，徹底杜絕舊年度文章：

| 層級 | 機制 | 效果 |
|------|------|------|
| L1：搜尋層 | Google News `after:{press_date}` | 從搜尋結果源頭過濾 |
| L2：候選層 | `is_after_press_date()` 硬性比對 | 有日期且早於 press_date → 直接淘汰 |
| L3：LLM 層 | Prompt 含 press_date + 年份意識 | 無日期但看起來是舊文 → false |

```python
def is_after_press_date(article_date, press_date_str):
    if not article_date:
        return None  # 無日期 → uncertain → 交給 LLM
    a = datetime.strptime(article_date[:10], "%Y-%m-%d").date()
    p = datetime.strptime(press_date_str, "%Y-%m-%d").date()
    return a >= p  # True=通過, False=淘汰
```

**實測效果**：
- 舊版一月報告含 43 篇 pre-press-date 文章 (18%)
- v4.1 一月報告：**0 篇** pre-press-date 文章

### 5.2 DDG snippet 日期提取

DDG 不直接提供日期，但 snippet (body) 常含日期資訊：

```python
def extract_date_from_snippet(body):
    patterns = [
        r'(\d{4})[/.-](\d{1,2})[/.-](\d{1,2})',  # 2026-01-05
        r'(\d{4})年(\d{1,2})月(\d{1,2})日',        # 2026年1月5日
    ]
    for pat in patterns:
        m = re.search(pat, body)
        if m:
            return f"{y}-{mo:02d}-{d:02d}"
    return ""
```

在送 LLM 前先排除明顯非文章的結果：

def is_likely_article(url, title, brand_keywords):

## 六、LLM 嚴格比對

### 6.1 核心原則

LLM 篩選是品質的關鍵。以下是經過多次迭代後的最佳實踐。

### 6.2 Prompt 設計（v4.2 — 品牌動態版）

v4.2 引入 `BRAND_PROMPT_MAP`，根據 `--brand` 參數動態生成 prompt，避免全家 prompt 用在 7-ELEVEN 任務上導致誤判。

```python
BRAND_PROMPT_MAP = {
    "familymart": {
        "brand_name": "全家便利商店（FamilyMart）",
        "competitor_list": "7-11、統一超商",
    },
    "7-11": {
        "brand_name": "7-ELEVEN（統一超商）",
        "competitor_list": "全家、FamilyMart",
    },
}
```

**動態 prompt 範例（7-ELEVEN）**：
```
你是嚴格的新聞露出檢核員。以下是搜尋「{task_name}」（7-ELEVEN（統一超商）的活動/產品）的候選結果。
本次新聞稿發稿日為：{press_date}

✅ 相關：文章主題是在報導 7-ELEVEN（統一超商）的「{task_name}」活動或商品
❌ 不相關：其他品牌（全家、FamilyMart）的活動，沒提到 7-ELEVEN
❌ 不相關：文章日期在 {press_date} 之前（往年同名活動的報導）
⚠️ 注意：日期「未知」但內容看起來是 2026 年新聞稿的露出 → 判 true

回答 JSON array: [{"id": 1, "relevant": true/false, "type": "原生"/"轉載", "reason": "簡短理由"}]
```

**v4.2 關鍵改進**：
- **品牌動態化**：prompt 中的品牌名、競品名自動依 `--brand` 切換
- 明確提供 `press_date`，讓 LLM 有日期參照基準
- 對無日期文章給予合理空間（標題像新文章 → pass）
- 對年份詞彙敏感（2024、2025、去年 → reject）

### 6.3 關鍵設計要素

| 要素 | 為什麼重要 |
|------|-----------|
| **提供新聞稿具體內容** | 只給標題不夠，LLM 需要知道具體的產品/活動/事件才能精準判斷 |
| **列出常見誤判** | 明確告訴 LLM「福袋不等於 LuLu 豬」「開工優惠不屬於貓之日」 |
| **提供發稿日** | 讓 LLM 能判斷日期合理性 |
| **區分有內文 vs 只有標題** | Google News 文章無法抓內文，需容許標題判斷 |
| **允許「泛泛整理文」pass** | 媒體優惠懶人包有提到具體產品應該算 |

### 6.4 常見誤判模式（must_not）

```python
# 全家 LuLu 豬新聞稿
"must_not": ["福袋", "家樂福", "貼圖", "LINE"]
# → 避免撈到：家樂福 LuLu 聯名、LINE 拜年貼圖、新年福袋

# 全家貓之日新聞稿
"must_not": ["開工", "咖啡買一送一"]
# → 避免撈到：開工優惠文章被關鍵字「全家」誤匹配

# 全家情人節新聞稿
"must_not": ["開工", "228", "連假"]
# → 避免撈到：228 連假優惠、開工優惠
```

- 含內文比對：每批 30 篇（內文佔 token）

## 八、回補機制

嚴格篩選後容易過度移除有效文章。需要回補機制。

### 8.1 回補流程

```
v_full（完整版，寬鬆篩選） → v_strict（嚴格篩選版）
                                    ↓
                            找出「在 v_full 但不在 v_strict」的文章
                                    ↓
                            用較寬鬆的 LLM 重新判斷
                                    ↓
                            通過的合併回 v_strict
```

### 8.2 寬鬆 LLM Prompt（回補用）

```
判斷標準（較寬鬆，但仍需相關）：
1. 標題主題與新聞稿有明確關聯 → pass
2. 泛泛的優惠整理文但有提到此主題 → pass
3. 報導同品牌同主題但角度不同（如評論、整理）→ pass
4. 完全不同主題 → fail
5. 中國/購物/食譜/非新聞 → fail
```

### 8.3 原則

- **每個新聞稿都必須有對應的文章**，不會有 0 篇的情況
- 若嚴格篩選後某個任務歸零，必須從寬鬆版回補
- 回補時使用比嚴格版寬鬆、比完整版嚴格的中間標準

## 十一、踩過的坑

### 11.1 URL 去重太激進

**問題**：移除所有 query 參數（`?p=12345`），導致 WordPress 文章被合併
**解法**：只移除已知的追蹤參數（`utm_*`, `fbclid`），保留其他

### 11.2 DuckDuckGo 回傳舊文

**問題**：搜「全家 情人節 玫瑰花」會回傳 2025 年七夕的文章
**解法**：對直連 URL 抓取網頁驗證年份；DDG 文章日期不可信任

### 11.3 LLM 只看標題，標題相似但主題不同

**問題**：「全家 開工優惠」文章被分到「貓之日」因為都含「全家」
**解法**：LLM prompt 中明確列出 must_not 排除條件

### 11.4 Google News URL 無法抓取內文

**問題**：`news.google.com/rss/articles/...` 是 JS 重定向，httpx 無法跟隨
**解法**：接受無法抓取，對 Google News 文章用標題+媒體名做判斷

### 11.5 嚴格篩選導致某些任務歸零

**問題**：BOYNEXTDOOR、齋戒月等較冷門主題被過度篩除
**解法**：回補機制 — 從寬鬆版找回有效文章，確保每個任務都有結果

### 11.6 轉載來源容易混入無關內容

**問題**：蕃新聞、鉅亨網等轉載站文章標題可能與原始內容不符
**解法**：對轉載來源加強 LLM 驗證，或在 prompt 中額外標注

### 11.7 發稿日前出現報導

**問題**：情人節新聞稿 2/9 發，但出現 2/2 的文章
**解法**：硬性日期過濾 — `article_date >= pr_date - 1 day`

### 11.8 同一錯誤標題在多家媒體重複出現

**問題**：一篇錯誤匹配的文章被多家媒體轉載，全部一起混入
**解法**：LLM 每批處理要連標題都一起看，發現相同標題就統一判定

## 十三、同一新聞稿多次發稿（拆分處理）

### 13.1 情境

某些促銷活動（開工優惠、節日活動等）會在不同日期發布內容相似但細節不同的新聞稿。客戶要求「拆開計算前後的露出成效」。

### 13.2 辨識方法

- 客戶新聞稿清單中出現標題相似但日期不同的多筆
- 同一掃描任務中的文章跨越不同發稿日期群組

### 13.3 拆分流程

```
1. 確認發稿次數和日期（從客戶新聞稿清單取得）
   ├── 第一次：例如 2/11
   └── 第二次：例如 2/20

2. 從明細報表中按發布日期切割文章
   ├── 發稿日1 ~ 發稿日2前一天 → 歸入第一次發稿
   └── 發稿日2 以後 → 歸入第二次發稿

3. 分別計算統計數據
   ├── 第一次：news_ch / news_art / social_ch / social_posts
   └── 第二次：news_ch / news_art / social_ch / social_posts

4. 在統計報表中拆為兩行，加註區分
```

### 13.4 實例

**全家開工優惠**（2026 年 2 月）：
- 原始：1 筆合併，31 頻道 / 53 篇
- 拆分後：
  - 第一次發稿 2/11：5 頻道 / 7 篇，9 社群頻道 / 71 篇
  - 第二次發稿 2/20：26 頻道 / 46 篇，60 社群頻道 / 469 篇

**7-ELEVEN 開工優惠**：
- 不同標題的兩篇新聞稿被合併計算
- 拆分為「推開工優惠」（33/59）和「開工咖啡優惠」（13/19）

## 十九、踩過的坑（v4.2 新增）

### 19.1 品牌 prompt 硬寫全家導致 7-ELEVEN 誤判

**問題**：LLM prompt 寫死「全家便利商店」，掃 7-ELEVEN 時把真正相關的文章判 false
**解法**：引入 `BRAND_PROMPT_MAP`，根據 `--brand` 動態生成 prompt

### 19.2 combined JSON 結構不一致

**問題**：手動合併多個 task 的掃描結果後，JSON 從 `{"results": [...]}` 變成 `{"taskA": {"results": [...]}, "taskB": {...}}`，`cmd_monthly` 用 `v4["results"]` 直接 crash
**解法**：`cmd_monthly` 改為同時支援兩種結構，自動偵測並 flatten

### 19.3 DDG 搜尋在 7-ELEVEN 品牌上效果偏低

**問題**：7-ELEVEN 品牌名有多種寫法（7-ELEVEN / 7-11 / 統一超商 / 711），DDG 搜尋若只用 4 組關鍵字覆蓋率不足
**解法**：
- 每個 task 擴展到 8–13 組關鍵字，覆蓋所有品牌名寫法
- DDG `max_results` 從 50 → 80
- Phase 1b 使用前 6 組關鍵字（原本 4 組）
- Phase 1c timeout 從 120s → 300s

### 19.4 作者欄位重複媒體名

**問題**：初始版本 fallback 直接用媒體名當作者，客戶回報「作者都跟媒體名一樣」
**解法**：開發 `extract_authors.py` 從文章 HTML 爬取真實作者，並清洗掉與媒體名相同的結果

### 19.5 統計表與明細表數字不一致

**問題**：統計表的「新聞總主文數」(439) ≠ 明細表總行數 (465)，因為部分文章被歸類為「社群」而未計入新聞
**解法**：取消新聞/社群分類（根據客戶要求），所有文章統一計入「新聞總主文數」
